# Pragmatist: multi-level versioning contract

**Target**: `docs/architecture/research/2026-05-10-multi-level-versioning-contract-{discovery,research}.md` (pre-ADR decision)
**Date**: 2026-05-10

## Summary

Контракт описан как «лёгкий, без enforcement, под solo-maintainer'а» — но в реальности это **семь независимых ручных дисциплин** на каждое изменение плагина (bump version в `plugin.json`, синхронизация README maturity-таблицы, решение «breaking или нет» по 4-пунктовому списку, написание миграционного файла, обновление CHANGELOG, не-bump marketplace, не-добавление `dependencies`). Через 6 месяцев на двух плагинах, при cadence «правлю по выходным», часть из них тихо отдрейфует — потому что **ни один пункт не self-enforcing, и ни один разрыв контракта не бьёт по поверхности**, пока внешний пользователь не запустит `/upgrade` через год после init и не получит missing migration. Steady-state — не катастрофа, но контракт даёт maintainer'у чек-лист, который он будет помнить ровно первые 2-3 bump'а.

## Operational findings

### P-1: Версия в трёх местах, ни одно не source-of-truth

**Category**: operational debt / hidden overhead

**The reality**: На каждый bump плагина Igor должен синхронно обновить: (1) `plugins/<role>/.claude-plugin/plugin.json` поле `version`, (2) `CHANGELOG.md` плагина с правильной семантикой (major/minor/patch), (3) README таблицу suite-level — колонка status «active (v1.0)» / «scaffolded (v0.1)», которая по контракту является proxy для semver. Три места, нет single source — значит **первое же расхождение** (например, bump `product` 0.1.0 → 0.2.0 в `plugin.json` без обновления README) silent. Для пользователя видна несоответствующая версия в `/plugin list` vs README → доверие падает, но maintainer об этом узнаёт только когда кто-то отрепортит. Без CI script, который грепает три места и сравнивает — этот дрейф **гарантирован** через 4-5 bump'ов. Контракт явно не называет, какое из трёх мест выигрывает при конфликте.

**Cost estimate**: ручное обновление 3 мест × ~5 bump'ов в год × 2 плагина ≈ 30 точек синхронизации. Realistic miss-rate без чеклиста: 15-25%.

**Severity**: medium — chronic friction, не выстрелит outage'ом, но эродирует доверие к версии как к контракту.

### P-2: «Breaking change» — list из 4 пунктов, который Igor должен помнить через год

**Category**: operational debt / on-call burden

**The reality**: Контракт фиксирует «breaking = rename/remove contribution points (commands, skills, agents, hooks) ИЛИ change input schema commands ИЛИ change frontmatter contract». Звучит чётко в момент чтения research-digest. Но через 8 месяцев, в 23:00 в воскресенье, когда Igor делает «маленький cleanup» в `architect/skills/role/SKILL.md` и rewording'ит system prompt — он **не вспомнит**, что у него есть формальный список и что нужно проверить «затрагивает ли это `frontmatter contract` артефактов». Скорее всего, он просто не bump'нет, и через ещё 3 месяца, после 2-3 таких «small cleanups», накопится фактически breaking-delta без version bump. Внешний пользователь, обновившийся через год, получит «стабильную 1.0.x» с поведением, которое не соответствует тому, что у него лежит в `docs/`. На on-call языке: **silent semver corruption**. Это не paged-event, потому что pages нет — это «когда-нибудь придёт issue с непонятной фрустрацией».

**Cost estimate**: вероятность miss'а на одном rewording-сессии ~30%; за год ~5 таких сессий → 1-2 невидимых breaking-violations накапливается. Когда выстрелит — единичный issue, ~2-4 часа на debug «что я когда сломал».

**Severity**: high — это эрозия именно того, что контракт обязан гарантировать. Если semver lies, дальше bump'ы тоже lies.

### P-3: `.architect-version` / `.product-version` маркер vs migrations stub

**Category**: deployment risk / hidden overhead

**The reality**: По контракту Q4=a маркер ровно повторяет `plugin.json.version` после `init` или `upgrade`. Но: `product/commands/upgrade.md` сегодня — stub, который «reports no migrations to run; updating .product-version». Это значит что **первый же реальный breaking-change в product** (например, при v0.1 → v1.0 на content-fill) поставит maintainer'а перед выбором: либо написать миграцию задним числом (но `migrations/` directory ещё ни разу не использовался — формат не калиброван), либо выпустить bump без миграции и сломать чужие проекты. `architect/commands/upgrade.md` уже более продвинут (читает CHANGELOG, gates по version), но и там migration-list зашит в сам command-file (`### 0.2 → 0.3`, `### 0.3 → 0.4 (placeholder)`) — то есть **миграции живут не как файлы в `migrations/`, а как inline-секции в `upgrade.md`**. Контракт говорит «миграции в `plugins/<role>/migrations/NNNN-from-X.Y.Z-to-A.B.C.md`», `architect` уже не следует этому. Расхождение между декларацией и реальностью даже на reference-плагине. Когда Igor через 4 месяца сядет писать первую реальную миграцию для `product`, он 30-40 минут потратит просто на «как мы вообще договорились» (контракт говорит одно, architect делает другое).

**Cost estimate**: первая реальная миграция = 2-4 часа дополнительного yak-shaving из-за нерешённого формата. Вероятность что Igor выберет «пропущу миграцию, всё равно один пользователь» ≈ 60%.

**Severity**: medium-high — конкретный известный долг с конкретной датой prod-impact (первый bump product за 0.1.x).

### P-4: `marketplace.json.version` — поле, которое контракт оставил «жить как захочет»

**Category**: operational debt / hidden overhead

**The reality**: Контракт фиксирует «bump только когда меняется структура `marketplace.json` или политики marketplace», и «после ADR-0001 структура не изменилась → остаётся 0.1.0». Это логически чисто, но операционно опасно: **Igor должен помнить отрицательное правило** — «не bump'ай это поле просто потому, что что-то изменилось в репо». Через 6 месяцев, когда он добавит `ops` плагин в `marketplace.json` (новая запись в массиве `plugins[]`), возникнет вопрос — это «структура marketplace.json» или «контент»? Контракт даёт ответ для добавления плагина (нет, не структура), но это требует mental-trace каждый раз. Альтернативно: Igor ради маркетинга решит bump'нуть `marketplace 0.1.0 → 1.0.0` при выходе ops, и контракт нарушен молча. **Никто не заметит** до момента, когда будущий пользователь увидит несоответствие «marketplace 1.0 но в нём scaffolded плагин 0.1» и спросит. Поле — формальность для schema-conformance, но контракт нагрузил его смыслом, который требует дисциплины поддерживать.

**Cost estimate**: 1-2 минуты mental-trace на каждое значимое изменение marketplace.json (~6 раз в год). Вероятность дрейфа от формальной семантики ≈ 40% за 12 месяцев.

**Severity**: low-medium — annoyance, livable, но иллюстрирует pattern «контракт описывает то, что не нужно делать, без механизма не делать».

### P-5: Bus factor = 1 для интерпретации контракта

**Category**: skills & bus factor

**The reality**: Контракт держится **в голове у Igor'а**. Research-digest и discovery лежат в `docs/architecture/research/`, но интерпретация «является ли вот это конкретное изменение breaking» — нигде не зафиксирована операционно (нет, например, decision-log где Igor пишет «bump product 0.1.0 → 0.2.0; рассмотрел: rename ✗, schema change ✗, frontmatter ✗ → minor»). Если Igor через 3 месяца уйдёт на 4 недели в отпуск (или на FTE, или просто болеет), и кто-то — реальный contributor или Future-Igor с потерянным контекстом — захочет сделать version bump, ему придётся: прочитать ADR (которого ещё нет, в процессе написания), прочитать research-digest на 70 строк, прочитать discovery на 150 строк, и **самостоятельно соотнести diff с критериями**. Это не сломанный процесс, это **отсутствие процесса**. Контракт правильно говорит «minimum overhead для solo», но не отвечает «как этот solo держит коntract сам с собой через 8 месяцев простоя».

**Cost estimate**: re-onboarding в собственный контракт через 6+ месяцев простоя ≈ 1-2 часа на каждую decision-point. Вероятность что Future-Igor просто bump'нет «по ощущению» вместо проверки ≈ 70%.

**Severity**: medium — не выстрелит, но накапливающийся дрейф quality.

### P-6: Что замечает дрейф? Никто и ничто

**Category**: on-call burden / hidden overhead

**The reality**: Перечислим failure modes контракта и кто их ловит:

| Drift | Кто заметит | Когда |
|---|---|---|
| `plugin.json.version` не bump'нут после rename command | никто | пока внешний пользователь не запустит `/upgrade` |
| README maturity-таблица отстаёт от `plugin.json.version` | никто | пока кто-то не сравнит вручную |
| `marketplace.json.version` неправомерно bump'нут | никто | возможно никогда |
| `.architect-version` маркер записан с ошибочным форматом | никто (ни schema, ни hook не валидируют) | при первом `upgrade` |
| Breaking change без bump'а major | никто | при поломке у пользователя |
| `dependencies` field случайно добавлен (нарушение Q7=c) | Claude Code загрузит и попытается резолвить | при install у пользователя |
| Миграция написана, но не запускается из-за неправильного naming | никто | при первом реальном `upgrade` |

Шесть из семи — **silent failures**. Это нормально для solo-проекта, но контракт об этом ни в одном месте честно не говорит «принимаем что enforcement = none, надеемся на дисциплину». Звучит как defended контракт, ведёт себя как honor system. Discrepancy между tone и реальностью — само по себе debt: maintainer верит, что у него есть контракт, и принимает решения исходя из этой веры, а не из реального уровня защиты.

**Cost estimate**: ноль на старте, ~2-6 часов debug'а на каждый случай когда кто-то наткнулся.

**Severity**: high — это headline operational reality, которую proposal не признаёт явно.

### P-7: Минимальный self-defense, который вписывается в «no CI» constraint

**Category**: tooling & ergonomics

**The reality**: Не предлагаю CI и не предлагаю pipeline (это уход в архитектуру, не моя роль). Но операционно — **один bash-скрипт `scripts/check-versions.sh`**, который грепает 3 места (plugin.json, CHANGELOG последний H2, README таблицу) и orfaюnt diff если они расходятся, — превратит P-1 из «дрейф через 5 bump'ов» в «дрейф никогда, потому что я запускаю это перед commit». Это не CI (никто не enforce'ит), это **локальный smoke-тест**, который соответствует ARCHITECTURE.md §4 «git + Claude Code как runtime, edit → /reload-plugins». Контракт сам по себе чистый, но без этого крошечного tooling-штриха он не feasible для steady-state. Аналогично — простой `scripts/breaking-check.sh`, который diff'ит `commands/*.md` filenames между двумя git-tags и предупреждает «вот эти команды исчезли — это breaking?». Не блокирует, не gate, просто напоминалка перед bump'ом.

Что важно: контракт **сейчас этого не предусматривает**, и без этого Igor через 4 месяца окажется ровно в P-1 / P-2 / P-6.

**Cost estimate**: 1-2 часа написать оба скрипта. Окупится после первого предотвращённого расхождения.

**Severity**: low (gap), но критическая mitigation для P-1, P-2, P-6.

## What's understated in the proposal

- **«Solo maintainer + STRATEGY §1 «без команды на найм пока». Минимум overhead. Контракт должен быть запоминаемым в голове, не требовать tooling для enforce.» (Forces F5).** Это самая опасная фраза. Она звучит как «мы выбрали лёгкость», а на деле говорит «мы согласны на silent corruption». В голове запоминается hypothesis, не contract. Контракт — это то, к чему можно вернуться через 6 месяцев. «Запоминаемое в голове» через год становится «не помню как мы договаривались, сделаю как кажется правильным сейчас».

- **«Простота > концептуальная чистота.» (Initial bias на Q4).** Простота записи — да. Но простота поддержания — обманчива. Маркер, который ровно повторяет `plugin.json.version`, кажется простым, **до момента когда есть три места правки и одно из них не обновили**. Простота на write-side = сложность на check-side, если check-side не существует.

- **«В v0.1 миграций нет — пока только tactical» (ARCHITECTURE.md §6 о миграциях).** «Пока tactical» — это «ещё не написал», но контракт уже зависит от того, что миграции работают. `architect/upgrade.md` уже описывает 0.2 → 0.3 миграцию. Это **не tactical**, это уже active surface, просто не названный.

- **«Каждый плагин эволюционирует независимо.» (Q2=a).** Звучит освобождающе, но игнорирует, что **мета-форма единая**. Когда меняется frontmatter contract в `architect-conventions`, и `product-conventions` копирует тот же contract — кто bump'ает кого? Контракт говорит «у того, кого изменение реально затрагивает» (Q3=d-гибрид из research). Но мета-форма затрагивает обоих по определению. Дилемма не решена, она замаскирована словом «независимо».

- **«Без CI и review — кто/что заметит drift?»** — этот вопрос в самом запросе на прожарку. Контракт сам на него **не отвечает**. Это не понравилось бы futurist'у; мне как pragmatist'у не нравится, что отсутствие ответа не задокументировано как принятый риск.

## What's missing entirely

- **Process для самой операции bump'а.** Шаг-за-шагом: что Igor делает физически когда решает bump. Открывает plugin.json — меняет — открывает CHANGELOG — пишет — открывает README — обновляет — git commit. Это не описано. Без чек-листа, контракт — теория.

- **Rollback plan для bump'а.** Что если bump'нул и понял через час, что был не прав (например, посчитал change minor, оказался breaking)? Tag уже выпущен. Pull от пользователя уже мог случиться. Контракт молчит.

- **Cost-ceiling / signal что overhead уже превысил порог.** Когда контракт перестаёт «весить ноль» и начинает съедать subjektивно много часов в квартал, какой триггер заставляет Igor его пересмотреть? Нет review-cadence, нет «через 6 месяцев пересчитаем». Контракт claim'ает «sticky» — но sticky без revisit-механизма = окаменелость.

- **Что делает `init` если маркер уже существует с не-той версией.** Контракт говорит «маркер записывается ровно plugin.json.version на момент init». Но init по семантике idempotent. Если пользователь вручную правил маркер (зачем-то), `init` его перезапишет? Спросит? Не упомянуто.

- **Релиз-cycle как отдельный артефакт.** Тут вообще нет понятия «релиз». Есть `version` в `plugin.json`, есть git-история. Нет git-tag convention (хотя research-digest упомянул `{plugin-name}--v{version}` как Claude Code-supported). Контракт не фиксирует tagging — значит каждый bump — это голый commit, без recoverable «вот была версия 1.0.0, она была вот этим коммитом».

- **CHANGELOG как source-of-truth (вместо README maturity-таблицы).** В `architect/` уже есть полноценный CHANGELOG.md в Keep-a-Changelog формате. В `product/` — нет. Контракт молча подразумевает, что CHANGELOG обязателен для каждого плагина, но не назначает его как «вот **здесь** ровно живёт версия и её семантика» (что разрешило бы P-1 — README таблица становится generated, а не вторая truth).

## What's actually realistic

70%-версия контракта, которая **встанет в production у solo-maintainer без сюрпризов**:

1. **Pin'ани одно место как source-of-truth: `plugins/<role>/.claude-plugin/plugin.json` поле `version`.** Всё остальное — производное. README таблицу обновляй ручно, но прими что она может отстать на пару недель — это не контракт, это маркетинг.
2. **Принимай Q1=c (marketplace.json.version статичный навсегда, например 1.0.0).** Не нагружай поле смыслом, который ты не хочешь поддерживать. Любая семантика, которую ты держишь дисциплиной — лучше не держи.
3. **Для Q3 (что breaking) — зафиксируй один bullet, не четыре.** Например: «breaking = пользователь должен запустить `/upgrade` или его проект сломается». Это **наблюдаемое** правило, а не структурное (4 пункта в research — структурные, легко забыть применить). Один bullet = меньше surface для ошибки в 23:00 в воскресенье.
4. **Напиши одну страницу `RELEASING.md`** в корне marketplace или в каждом плагине: «Когда bump'аю — делаю А, Б, В. Когда release — делаю Г, Д.» 15 строк. Это и есть «контракт, к которому можно вернуться через 6 месяцев».
5. **Прими operational reality честно** в самом ADR: «enforcement = none; ловит дрейф = только пользователь bug-репортом или сам maintainer при ревью; revisit-cadence = квартально вместе с STRATEGY review.» Не делай вид, что контракт защищён — просто скажи, что он держится на дисциплине, и эта дисциплина имеет известный failure rate.
6. **Один маленький bash-скрипт для P-7** — не сейчас, но **до первого реального bump'а**. Поставь себе todo «перед v0.2 product — написать `check-versions.sh`».
7. **Миграции — формат зафиксируй до первой реальной миграции, не «когда понадобится».** Формат должен совпадать с тем, что уже делает `architect/upgrade.md` (inline-секции в upgrade.md), либо `architect/upgrade.md` нужно отрефакторить под отдельные файлы. Расхождение между декларацией и reference-имплементацией нужно закрыть **до** того, как `product` начнёт мигрировать.

Это не редизайн. Это «возьми тот же контракт, но честно назови его границы и добавь две страницы операционных деталей, которых сейчас нет».

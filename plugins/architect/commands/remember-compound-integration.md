# /archforge:remember-compound-integration

Writes a project-level instruction block that tells Claude — in every session, regardless of which plugin is active — how `archforge` and the EveryInc `compound-engineering` plugin coexist in this project.

The block becomes part of the project's persistent memory (the file Claude reads at session start). After running this command once, both plugins know how to interleave.

## Inputs

- Optional flag: `--target=AGENTS.md` (default) or `--target=CLAUDE.md`.
- Optional flag: `--lang=en` (English) or `--lang=ru` (Russian) or `--lang=auto` (default — detect).
- Project context: detect the presence of CE plugin and its artifacts; detect the working language.

## Steps

### 1. Detect Compound Engineering presence

Check for any of these signals:
- File `AGENTS.md` at repo root mentioning `compound-engineering` or `/ce-`.
- Directories `docs/brainstorms/`, `docs/plans/`, `docs/solutions/`, or `todos/`.
- A user-level CE install (cannot always be detected from the repo, but absence of the above directories is a strong signal CE hasn't been used here yet).

If CE signals are absent:
- Tell the user (in the detected language): "I don't see Compound Engineering artifacts in this project. I can still write the integration block — it'll be a no-op until you install CE. Continue?"
- If yes — proceed. If no — abort.

### 2. Detect working language

If `--lang=auto` (default):
- Inspect, in order: `STRATEGY.md`, `ARCHITECTURE.md`, the most recent ADR in `docs/architecture/decisions/`, README.md, README.ru.md.
- For each: count Cyrillic vs Latin characters in the body (ignore code blocks). The first file with a >70% majority sets the language.
- If no clear majority anywhere → fall back to English.
- State the detected language to the user before writing: "Detected language: Russian. Writing the Russian version of the integration block." This gives the user a chance to override with `--lang=en`.

If `--lang=en` or `--lang=ru` — use it directly.

### 3. Detect target file

- Default: `AGENTS.md` at repo root.
- If user passed `--target=CLAUDE.md`, use `CLAUDE.md` instead.
- If neither file exists — create the chosen target with a minimal header in the chosen language.
- If the target file exists and **already contains a section** with the marker `<!-- archforge × compound-engineering integration -->` — replace that section in place, do not duplicate.

### 4. Use the `compound-integration` skill

Load the `compound-integration` skill. It contains the canonical rules. The command's job is to emit a **condensed, machine-parseable summary** into the target file — not to dump the full skill content.

### 5. Write the integration block — English version (when `--lang=en`)

```markdown
<!-- archforge × compound-engineering integration -->
<!-- Generated by /archforge:remember-compound-integration. Re-run to refresh. -->
<!-- lang: en -->

## Architecture (archforge) × Compound Engineering integration

This project uses two complementary plugins:

- **`compound-engineering`** (EveryInc) — feature-level workflow: Brainstorm → Plan → Work → Review → Compound.
- **`archforge`** — architectural cycle: Discover → Research → Design → Decide → Document → Review.

They interleave; they do not run in parallel. Detailed rules live in the
`compound-integration` skill of `archforge`. Quick reference below.

### Combined workflow

/ce-ideate (optional)
  -> /ce-brainstorm
       -> [is the task architecturally significant?]
            - NO  -> /ce-plan
            - YES -> /archforge:cycle (Discover -> [Research] -> Design -> Decide -> Document)
                       -> /ce-plan (uses ADR + brainstorm as input)
                            -> /ce-work
                                 -> /ce-code-review
                                      -> /archforge:review (ADR conformance)
                                           -> /ce-compound
                                                -> [new architectural pattern?]
                                                     - YES -> /archforge:adr

### Architecturally-significant triggers (run archforge cycle before /ce-plan)

- New module / new top-level directory.
- New runtime dependency in package manifest.
- New external service (DB, queue, cache, third-party API).
- New protocol (introducing gRPC, WebSocket, SSE, etc.).
- Data model change that breaks existing consumers.
- Change in deployment topology.
- A pattern decision the team hasn't made before (auth, error model, retry, caching).
- User explicitly says "this is architectural".

### Non-architectural triggers (skip archforge, go straight to /ce-plan)

Bug fixes, new endpoints/components in established patterns, tests, refactors
inside an existing module, style/formatting, doc updates.

### Artifact ownership

| File | Owner |
|---|---|
| `ARCHITECTURE.md`, `STRATEGY.md` | archforge |
| `docs/architecture/{decisions,diagrams,research,reviews}/`, `decision-map.md` | archforge |
| `docs/{brainstorms,plans,solutions}/` | compound-engineering |
| `todos/` | compound-engineering |
| `AGENTS.md`, `CLAUDE.md` | shared |

No file is written by both plugins.

### Cross-references — required

- CE plans that touch architecture **must reference relevant ADRs by number**.
- New ADRs produced inside a CE task **should reference the brainstorm/plan doc** in their Context section.
- Compound learnings (`docs/solutions/`) **may reference ADRs**; if the learning is itself architectural, file an ADR too.

### Review duties — split

- `/ce-code-review` covers correctness, security, performance, maintainability, testing.
- `/archforge:review` covers ADR conformance and architectural smells (boundaries, coupling, layering).

Run both on substantial PRs. Run only `/ce-code-review` on small/non-structural PRs.

<!-- end archforge × compound-engineering integration -->
```

### 6. Write the integration block — Russian version (when `--lang=ru`)

```markdown
<!-- archforge × compound-engineering integration -->
<!-- Сгенерировано командой /archforge:remember-compound-integration. Перезапусти для обновления. -->
<!-- lang: ru -->

## Интеграция архитектурного цикла (archforge) и Compound Engineering

В проекте используются два дополняющих плагина:

- **`compound-engineering`** (EveryInc) — рабочий цикл уровня фичи: Brainstorm → Plan → Work → Review → Compound.
- **`archforge`** — архитектурный цикл: Discover → Research → Design → Decide → Document → Review.

Они чередуются, а не работают параллельно. Подробные правила — в скилле
`compound-integration` плагина `archforge`. Краткая шпаргалка ниже.

### Объединённый поток работы

/ce-ideate (опционально)
  -> /ce-brainstorm
       -> [задача архитектурно значима?]
            - НЕТ -> /ce-plan
            - ДА  -> /archforge:cycle (Discover -> [Research] -> Design -> Decide -> Document)
                       -> /ce-plan (на вход — ADR и brainstorm)
                            -> /ce-work
                                 -> /ce-code-review
                                      -> /archforge:review (соответствие ADR)
                                           -> /ce-compound
                                                -> [появился новый архитектурный паттерн?]
                                                     - ДА -> /archforge:adr

### Триггеры архитектурной значимости (запускать archforge-цикл перед /ce-plan)

- Новый модуль или каталог верхнего уровня.
- Новая зависимость в манифесте пакетов.
- Новый внешний сервис (БД, очередь, кэш, сторонний API).
- Новый протокол (введение gRPC, WebSocket, SSE и т.п.).
- Изменение модели данных, ломающее существующих потребителей.
- Изменение топологии развёртывания.
- Решение по паттерну, которого команда раньше не принимала (авторизация, модель ошибок, ретраи, кэширование).
- Пользователь явно говорит «это архитектурное решение».

### Неархитектурные триггеры (пропускаем archforge, сразу /ce-plan)

Багфиксы, новые эндпоинты или компоненты по уже сложившимся паттернам, тесты,
рефакторинг внутри границы существующего модуля, форматирование,
обновления документации.

### Владение артефактами

| Файл | Владелец |
|---|---|
| `ARCHITECTURE.md`, `STRATEGY.md` | archforge |
| `docs/architecture/{decisions,diagrams,research,reviews}/`, `decision-map.md` | archforge |
| `docs/{brainstorms,plans,solutions}/` | compound-engineering |
| `todos/` | compound-engineering |
| `AGENTS.md`, `CLAUDE.md` | общие |

Ни один файл не записывается обоими плагинами.

### Перекрёстные ссылки — обязательные

- Планы CE, затрагивающие архитектуру, **обязаны ссылаться на нужные ADR по номеру**.
- Новые ADR, появившиеся внутри задачи CE, **должны ссылаться на brainstorm/plan-документ** в секции Context.
- Compound-учения (`docs/solutions/`) **могут ссылаться на ADR**; если само учение архитектурное — заводи ADR отдельно.

### Разделение обязанностей при ревью

- `/ce-code-review` проверяет корректность, безопасность, производительность, поддерживаемость, тесты.
- `/archforge:review` проверяет соответствие ADR и архитектурные смрады (границы, связанность, слоистость).

Запускай оба на крупных PR. На небольших или неструктурных PR — только `/ce-code-review`.

<!-- end archforge × compound-engineering integration -->
```

### 7. Tell the user what happened

Output to chat (in the chosen language):

- Path of the target file.
- Detected/chosen language.
- Whether the block was inserted or replaced.
- A reminder: this block is loaded at every session start. To refresh after CE upgrade or language switch, re-run this command.
- Suggestion: "Try running `/ce-brainstorm \"<feature idea>\"` followed by `/archforge:cycle` — Claude will know how to chain them now."

### 8. Do not commit

Leave that to the user.

## Discipline

- **Idempotent.** Re-running this command on the same project replaces the existing block, never duplicates it. The HTML-comment markers are the boundary.
- **Language switch is supported.** Re-running with a different `--lang` replaces the prior block atomically — no mixed-language artifacts.
- **No edits to other parts of `AGENTS.md`/`CLAUDE.md`.** Only the bounded block is touched.
- **Don't proceed silently if CE is absent** — confirm with the user.
- **The block is reference, not gospel.** It documents the interleaving rules. The full reasoning is in the `compound-integration` skill.

## When to suggest re-running

If during a session you observe:

- A CE command name that doesn't match the block (e.g., a renamed `/ce-work-v2`).
- The user mentioning an upgrade to CE.
- Drift between the actual workflow and what the block describes.
- The project's working language has clearly shifted (e.g., the block is in English but new ADRs are in Russian).

Then proactively mention (in the project's working language): "the integration block in `AGENTS.md` may be out of date — re-run `/archforge:remember-compound-integration` to refresh."
